# Архитектура языка pandaC (Draft)

## 1. Философия и цели (Philosophy & Goals)
**pandaC** — это транслируемый язык программирования.
- **Синтаксис:** Python-like (отступы для блоков кода, динамический "внешний вид", удобные литералы массивов и словарей `{}`, `[]`).
- **Исполнение:** Трансляция (Code Conversion) кода pandaC в эквивалентный C++ код с последующей компиляцией бэкендом (Clang/GCC) в нативный бинарник.
- **Цель:** Совмещение удобства и скорости написания скриптов (как в Python/JS) с производительностью и мощью нативного C++.

## 2. Пайплайн компиляции (Compilation Pipeline)
Процесс превращения `.pc` файла в `.exe` делится на 5 строгих фаз:

### 1. Preprocessor (`src/compiler/preprocessor.cpp`)
- **Вход:** Исходный `.pc` файл.
- **Обязанности:** Убирает комментарии (строчные `#` и многострочные `/* */`). Нормализует пустые строки.

### 2. Lexer/Tokenizer (`src/compiler/lexer.cpp`)
- **Обязанности:** Разбивает текст на токены. Критически важный шаг: отслеживает пробелы в начале строк и генерирует искусственные токены `INDENT` и `DEDENT` для определения границ блоков (замена `{` и `}` из C-подобных языков).

### 3. Parser (Syntax Analysis) (`src/compiler/lexer.cpp` -> `fromString` & `parseStatement`)
- **Обязанности:** Читает токены и строит Абстрактное Синтаксическое Дерево (AST). Использует паттерн Recursive Descent с приоритетом операторов (Pratt parsing). Гарантирует корректность синтаксиса.

### 4. Code Generator (`src/compiler/codeconvertion.cpp` -> `toCppString`)
- **Обязанности:** Проходит по AST и генерирует валидный C++ код, делегируя все операции в стандартную библиотеку `pandaC.cpp`.

### 5. C++ Backend Compiler
- Вызов системного компилятора (`clang++`), линковка сгенерированного `4.cpp` с библиотекой `pandaC.cpp` и сборка бинарного файла.

## 3. Система типов и управление памятью (Memory & Typing)
Для того чтобы статически типизированный C++ вел себя как динамический Python, ядром языка выступает архитектура универсального хранилища `PandaVar`.

### 3.1. Динамическая обертка (PandaVar)
Все переменные в pandaC по умолчанию генерируются как тип `PandaVar`.
- Под капотом `PandaVar` использует `std::any` для скрытия реального типа данных.
- Он способен хранить: `int`, `double`, `bool`, `std::string`, а также коллекции `PandaCDict` и `PandaCList`.
- Имеет перегрузки операторов явного кастинга (например, `explicit operator int()`), делегирующие итераторы `begin()/end()` и умный метод доступа по индексу/ключу `pandac_getitem()`.

### 3.2. Коллекции (Словари и Списки)
Типы коллекций: `PandaCDict` (аналог dict) и `PandaCList` (аналог list).
- **Режим передачи:** По ссылке (Pass-by-reference semantics), как в Python.
- **Реализация C++:** Внутренняя память оборачивается в `std::shared_ptr`.
    - `PandaCList` хранит `std::shared_ptr<std::vector<PandaVar>>`.
    - `PandaCDict` хранит `std::shared_ptr<std::unordered_map<std::string, PandaVar>>`.
- Благодаря тому, что коллекции хранят внутри себя `PandaVar`, язык поддерживает любую глубину вложенности без парсинга строк (например, `dict["a"]["b"] = [1, 2, 3]`). При копировании `PandaCDict a = b` копируется только указатель на данные.

## 4. Соглашения по генерации C++ кода (C++ CodeGen Rules)
### При трансляции pandaC -> C++ выполняются следующие правила:
1. **Типы переменных:** Генератор по умолчанию выводит ключевое слово `PandaVar` для всех переменных, делегируя разрешение типов в Runtime.
2. **Функции:** Если программист не указывает тип возвращаемого значения, функция генерируется с возвращаемым типом `PandaVar`. По умолчанию в конец всех функций дописывается `return PandaVar();` (аналог `return None` в Python), чтобы избежать Undefined Behavior в C++.
3. **Логика True/False:** Литералы переводятся в `true`/`false`. Условия ветвлений (`if`, `while`) оборачиваются в вызов `pandac_bool(expr)`, чтобы динамически приводить `PandaVar` к сырому C++ `bool`.
4. **Стандартные функции Python:** Вызовы `print()`, `len()`, `int()`, `float()` транслируются в шаблоны `pandac_print()`, `pandac_len()`, `pandac_int()`.
5. **Операторы (Математика и Сравнения):**
   Сырые C++ операторы не используются во избежание неоднозначностей (Ambiguous overloads). Все операторы в AST конвертируются в функции-утилиты:
    * `a + b`  -> `pandac_add(a, b)` (поддерживает конкатенацию строк)
    * `a * b`  -> `pandac_mul(a, b)` (поддерживает дублирование строк `"А" * 3`)
    * `a // b` -> `pandac_int_div(a, b)`
    * `a ** b` -> `pandac_pow(a, b)`
    * `a < b`  -> `pandac_less(a, b)`
    * `a == b` -> `pandac_eq(a, b)`
      и так далее.

## 5. Известные ограничения и Roadmap (Future Work)
1. **Контроль ошибок:** Сейчас при ошибке парсера выводится `Unexpected token: =`. В будущем лексер должен хранить `line_number` и `column` для каждого токена, чтобы выдавать ошибки в стиле Rust/Clang с указанием строки и каретки в файле `.pc`.
2. **Модульная система (Импорты):**
   Зарезервированное слово для подключения модулей — `using` (например, `using pandaC`, `using Math`).
   Текущее состояние: `using` используется для подключения встроенных библиотек, исходный код которых жестко прошит в `.conf` файлах и подключается при компиляции.
   В будущем: необходимо расширить логику `using` (или добавить `import`), чтобы позволить подключать пользовательские `.pc` файлы из текущей директории для сборки многофайловых проектов.
